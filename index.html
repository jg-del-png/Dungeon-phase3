How state is saved & restored: The app keeps its current loot list and party size in JavaScript variables. Whenever those values change, the app calls a function that converts the state to a string and writes it into localStorage. On page load, the app reads that string back from localStorage, converts it back into JavaScript objects, and uses those values as the starting state.

Why updateUI() must run after restoration: The UI is rendered from the in-memory state variables. If the DOM is updated before restoring saved state, the screen will show the default (empty) view even though the app later updates its internal state. Running updateUI() after restoration guarantees the displayed list, totals, and party size match the restored state.

What JSON.stringify() and JSON.parse() do: JSON.stringify() turns JavaScript values into a JSON string so it can be stored as text in localStorage. JSON.parse() takes that string and rebuilds JavaScript objects and arrays from it. Together they let the application persist structured data across reloads.

What we observe when stepping through restoration: When you set breakpoints in the restoration logic, you can see the app read raw text from localStorage, convert it with JSON.parse(), validate the shape of the data, and then assign it into the running variables. The next line usually calls updateUI(), and you can then see the DOM update to reflect the restored state.

How this supports the course outcomes: This phase demonstrates the difference between transient runtime state and persistent storage, and it shows how an app can manage its lifecycle without losing user data. It also reinforces good practices like validating loaded state and keeping UI rendering separate from storage logic.
